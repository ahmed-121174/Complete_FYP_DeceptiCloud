================================
1. PROJECT OVERVIEW
================================

DeceptiCloud is an AI-Driven Cyber Deception System that combines machine learning,
honeypot technology, and real-time traffic analysis to detect, classify, and deflect
cyber attacks. The system deploys 14 identical-looking websites (7 real + 7 honeypot
clones) behind an intelligent routing proxy that classifies incoming traffic using
ML models and rule-based analysis. Malicious traffic is transparently redirected to
honeypot clones while legitimate users access real services.

Key Innovation: Unlike traditional honeypots, DeceptiCloud uses Generative AI (LLMs)
to create dynamic, context-aware responses and Generative Adversarial Networks (GANs)
to populate databases with realistic synthetic identities. This makes the deception
indistinguishable from reality while fingerprinting attackers behaviorally.

│ RULE-BASED DETECTION ENGINE (Proxy)                                                 │
├─────────────────────────────────────────────────────────────────────────────────────┤
│                                                                                     │
│ In addition to ML models, the Routing Proxy uses real-time regex-based pattern      │
│ matching for immediate threat response, no ML latency. Patterns include:            │
│                                                                                     │
│ • SQL Injection Patterns:   UNION SELECT, OR 1=1, DROP TABLE, etc.                 │
│ • NoSQL Injection Patterns: $gt, $ne, $regex, {$where: ...}                        │
│ • XSS Patterns:             <script>, onerror=, javascript:, etc.                  │
│ • Path Traversal Patterns:  ../, ....//,  /etc/passwd, /etc/shadow                 │
│ • Suspicious User-Agents:   sqlmap, nikto, nmap, dirbuster, etc.                   │
│                                                                                     │
│ Scoring: Each pattern match adds to a threat score (0.0 to 1.0).                   │
│ If score >= 0.3, traffic is classified as malicious.                                │
│ Borderline cases (0.1-0.3) are sent to the ML API for confirmation.                │
│                                                                                     │
│ Known Attacker System: After 3+ attacks from same IP, all subsequent               │
│ requests are automatically routed to honeypots with 99% confidence.                │

7. DYNAMIC HONEYPOT ARCHITECTURE
================================

The Dynamic Honeypot system is designed to waste attacker time, gather intelligence,
and protect real assets through three key mechanisms: Deception, Isolation, and Dynamism.

7.1 Architecture & Purpose
──────────────────────────

  A. IDENTICAL UI/UX (The "Mirror" Strategy)
     Purpose: prevent attackers from realizing they are in a trap.
     Implementation:
     • Real site and Honeypot share the exact same HTML/CSS templates.
     • Differences are only in backend data (e.g., lower prices in honeypot to tempt attackers).
     • Result: An attacker manually browsing cannot distinguish the two.

  B. DYNAMIC ROTATION (The "Moving Target" Defense)
     Purpose: disrupt automated scanning and reconnaissance tools.
     Implementation:
     • Every 60 seconds, the honeypot "personality" rotates.
     • A honeypot might look like a Bank at 10:00 AM, and an E-commerce site at 10:01 AM.
     • This confuses scanners (e.g., Nmap) that expect static services.

  C. BEHAVIORAL TRACKING & BLACKLISTING
     Purpose: permanently neutralize detected threats.
     Implementation:
     • Proxy tracks the source IP of every confirmed attack.
     • If IP attacks > 3 times:
       - IP is permanently flagged as "Known Attacker".
       - ALL future traffic from this IP is routed to honeypots, even for benign requests.
       - The attacker thinks they are hacking the system, but are only interacting with decoys.

  D. FAKE RESPONSE HEADERS
     Purpose: misinform attackers about the technology stack.
     Implementation:
     • Honeypots inject random headers like `Server: Apache/2.4` (when running Flask)
     • or `X-Powered-By: PHP/7.4` (when running Python).
     • This provokes attackers to launch wrong exploits, wasting their resources.

8. ROLE OF DOCKER
================================

Docker provides containerization for the application:

What Docker Does:
  • Packages each component (ML API, Proxy, Dashboard, 14 websites) into
    isolated containers with all their dependencies
  • Ensures consistent environments across development, testing, and production
  • Eliminates "works on my machine" problems

Docker Compose Configuration (docker/docker-compose.yml):
  • Defines 17 services on a shared 'cyberdefense' network
  • Each website runs in its own container with environment variables
    (SITE_TYPE, SITE_VARIANT, IS_HONEYPOT, SITE_PORT)
  • Volumes mount model files and log directories
  • Containers can communicate via service names (e.g., 'ml-api:5000')

Dockerfiles Created:
  • ml-api.Dockerfile      — TensorFlow + Flask for ML model serving
  • proxy.Dockerfile       — Flask proxy with requests library
  • website.Dockerfile     — Generic Flask website (one image, 14 containers)
  • dashboard.Dockerfile   — Dashboard Flask app with Chart.js

How to Deploy:
  cd docker/
  docker-compose up -d              # Start all 17 services
  docker-compose logs -f proxy      # View proxy logs
  docker-compose down               # Stop everything

9. ROLE OF KUBERNETES (K8s)
================================

Kubernetes provides production-grade orchestration:

What Kubernetes Does:
  • Manages deployment, scaling, and operations of containerized applications
  • Provides automatic scaling, self-healing, and rolling updates
  • Implements network policies for security isolation

Kubernetes Configuration (kubernetes/cyber-deception.yaml):

  RESOURCE TYPE          WHAT IT DOES
  ─────────────────────  ──────────────────────────────────────────────────
  Namespace              Creates isolated 'cyberdefense' namespace
  ConfigMap              Stores shared config (ML API URL, rotation interval)
  Deployment (ML API)    Runs ML model server with health probes
  Deployment (Proxy)     Runs routing proxy with auto-restart
  Deployment (Dashboard) Runs dashboard with LoadBalancer service
  StatefulSet (Sites)    Runs 14 websites with persistent storage
  Service (ClusterIP)    Internal service discovery (ml-api, sites)
  Service (LoadBalancer) External access (proxy:8080, dashboard:9000)
  NetworkPolicy          Isolates honeypots from real sites at network level

Key Kubernetes Features Used:
  • Health Probes: Liveness and readiness checks on all services
  • Network Policies: Prevents honeypots from accessing real site data
  • StatefulSets: Ensures each website has persistent database storage
  • Resource Limits: Memory and CPU limits per container
  • Rolling Updates: Zero-downtime updates when changing configurations

How to Deploy:
  kubectl create namespace cyberdefense
  kubectl apply -f kubernetes/cyber-deception.yaml
  kubectl get pods -n cyberdefense    # Check status
